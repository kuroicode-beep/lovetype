# LoveType 작업 기록

## 1. 저장소 준비

- `C:\Projects\lovetype` 폴더가 처음에는 Git 저장소가 아닌 상태임을 확인
- 로컬 Git 저장소 초기화
- 기본 브랜치를 `main`으로 변경
- 원격 저장소 `https://github.com/kuroicode-beep/lovetype.git` 연결
- 첫 커밋 생성 및 `origin/main`으로 푸시

## 2. 첫 랜딩 화면 구현

- 정적 페이지 구조로 `index.html`, `styles.css`, `script.js` 생성
- GitHub Pages에 바로 올릴 수 있도록 상대경로 기반으로 구성
- 모바일 우선 레이아웃으로 랜딩 화면 구현
- 상단 헤더에 `LoveType` 로고 영역과 고대비 모드 스위치 추가
- 히어로 영역에 메인 카피, 서브 카피, 메타 정보, 메인 CTA 배치
- 이미지 없이 CSS 기반 심볼 비주얼 구현
- `mode=hc` URL 파라미터와 `localStorage`를 이용한 테마 전환 구조 구현
- 포커스 상태, 스킵 링크, 버튼 접근성 등 기본 접근성 반영

## 3. 원페이지 흐름으로 확장

- 단일 페이지 안에서 `랜딩 -> 문항 -> 결과` 화면 전환 구조로 개편
- 문항 화면 추가
- 결과 화면 추가
- 결과 화면에 MBTI 코드, 강도 설명, 축 그래프, 총평, 연애 스타일, 조심할 포인트, 공유 버튼, 재테스트 버튼, 다운로드 CTA 구성
- 오늘 완료 이력이 있으면 랜딩 대신 결과 화면으로 진입하는 재방문 처리 추가
- 1일 1회 재테스트 제한 UI와 문구 반영

## 4. 데이터 소스 전환

- 기존 데이터 파일 대신 `lovetype.json`을 기준 데이터셋으로 사용하도록 구조 변경
- `lovetype.json`의 `service`, `version`, `locale`, `totalQuestions`, `dimensions`, `questions[]` 구조 확인
- 스크립트에서 문항을 하드코딩하지 않고 `fetch("./lovetype.json")` 기반으로 로딩하도록 변경
- 파일 상단의 주석 라인을 제거한 뒤 JSON 파싱하도록 보완
- `totalQuestions` 값을 실제 데이터 기준으로 진행률과 문항 수 표시에 반영
- 각 문항의 `dimension`, `theme`, `question`, `options[2]`를 화면에 그대로 렌더링하도록 변경

## 5. 문항 진행 로직 구현

- 한 화면에 질문 1개만 표시하는 구조 유지
- 상단 진행률 바와 현재 문항 번호 표시 구현
- `theme`를 보조 텍스트로 노출
- 선택지 2개를 버튼형 카드로 렌더링
- 선택 즉시 응답을 저장하고 다음 문항으로 자동 이동하도록 구현
- 20문항 전체를 `lovetype.json` 순서대로 진행하도록 구성

## 6. 결과 계산 로직 구현

- `EI`, `SN`, `TF`, `JP` 각 축별로 점수 누적 구조 구현
- 각 선택지의 `score` 객체 값을 해당 축 점수에 반영
- 축별 우세 값을 조합해 최종 MBTI 4글자 산출
- 축별 점수 수치와 우세 방향을 결과 데이터로 저장
- 강도 규칙 적용
- 차이 `3 이상 = 강`
- 차이 `2 = 중`
- 차이 `1 또는 0 = 약`
- 결과 화면 강도 문구 예시 형식으로 출력
- `외향-중 / 직관-강 / 감정-중 / 인식-약`

## 7. 결과 화면 정교화

- 결과 화면 상단에 모바일 캡처에 적합한 결과 카드 배치
- MBTI 결과 코드 강조 표시
- 강도 설명 텍스트 추가
- 4축 그래프를 막대형 시각화로 구현
- 일반 모드와 고대비 모드 모두에서 식별 가능한 색 대비 적용
- 총평, 연애 스타일, 강점, 조심할 포인트 섹션 분리
- 공유 버튼과 다운로드 CTA를 하단에 배치

## 8. 결과 해설 데이터 구조화

- 초기에는 기본 placeholder 성격의 결과 설명 구조를 설계
- 이후 `descriptionMap` 형태로 MBTI 결과별 해설 매핑 구조 확장
- 최종적으로 16개 MBTI 전체에 대해
- `summary`
- `style`
- `strength`
- `caution`
- 값을 분리해 저장
- 향후 별도 결과 데이터 파일로 분리할 수 있게 구조 유지

## 9. 상태 저장 및 재방문 처리

- `localStorage`에 아래 값 저장 구조 구현
- `theme_mode`
- `completed_date`
- `mbti_result`
- `axis_scores`
- `result_summary`
- `result_detail`
- `next_available_at`
- 완료 여부는 KST 기준 날짜 문자열로 판단하도록 구현
- 오늘 이미 완료한 경우 결과 화면으로 바로 진입
- 다시 테스트하기 버튼 비활성화
- 안내 문구 표시
- `오늘의 테스트는 이미 완료했어요`
- `다시 테스트는 1일 1회만 가능합니다`

## 10. 공유 UX 구현
